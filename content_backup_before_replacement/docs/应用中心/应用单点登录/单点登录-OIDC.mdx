## 概述

<ImageZoom
  src="/images/应用中心/应用单点登录/单点登录-OIDC/overview.png"

OIDC（OpenID Connect）是基于 OAuth 2.0 协议之上的身份认证层，为应用提供验证终端用户身份的标准化方式。OIDC 已成为现代单点登录（SSO）系统的主流选择。

### 核心概念

| 术语 | 说明 |
|------|------|
| **身份提供商（IdP）** | 负责认证用户并提供身份信息的服务（如专属集成平台） |
| **服务提供商（RP）** | 依赖身份提供商进行用户认证的应用系统 |
| **ID Token** | 包含用户身份信息的 JWT |
| **Access Token** | 用于访问受保护资源的令牌 |
| **授权码** | 用于交换获取 ID Token 和 Access Token 的中间凭证 |
| **UserInfo 端点** | 提供额外用户信息的 API 端点 |

### OIDC 优势

- **标准化**：基于开放标准，兼容性好，易于集成
- **安全性**：使用 JWT 进行签名和加密，保证数据完整性和机密性
- **灵活性**：支持多种授权流程，适应不同应用场景
- **移动友好**：原生支持移动应用和单页应用

## 前提条件

- 拥有专属集成平台管理员账号
- 目标应用支持 OIDC 协议
- 确保专属集成平台中已创建相应用户

### 步骤 2：配置 OIDC 参数

{/* DOC_IMAGE src="/images/应用中心/应用单点登录/单点登录-OIDC/step-12.png" alt="配置 OIDC 参数截图" width="1200" height="720" */}

在应用详情页面，切换到**单点配置**选项卡，配置以下参数：

| 参数 | 说明 |
|------|------|
| 登录方式 | 控制访问该应用的登录方式选项 |
| 单点协议 | 选择 **OIDC** |
| RedirectURIs | 重定向 URI（需与应用回调地址一致） |
| 授权类型 | 推荐选择 `authorization_code` |
| 授权范围 | 通常包括 `openid`、`profile`、`email` 等 |
| ID Token 签名算法 | 推荐 RS256 |
| Access Token 过期时间 | 建议 1 小时 |
| Refresh Token 过期时间 | 建议 7 天 |

## 高级特性

### 授权流程类型

| 流程 | 适用场景 |
|------|---------|
| **授权码流程** | 服务器端应用（最常用、最安全） |
| **隐式流程** | 无后端的单页应用 |
| **混合流程** | 结合授权码和隐式流程的特点 |
| **密码凭证流程** | 高度信任的第一方应用 |
| **客户端凭证流程** | 服务器间通信，不涉及用户 |

### 安全措施

<Callout type="warn">
为保障 OIDC 实施安全性，请务必注意以下关键措施：
</Callout>

1. **PKCE**：防止授权码拦截攻击
2. **Nonce 值**：防止重放攻击
3. **State 参数**：防止 CSRF 攻击
4. **令牌加密**：保护敏感信息
5. **Client Secret 管理**：安全存储和定期轮换

## 常见问题

<Accordions>
<Accordion title="无法重定向到 IdP">
**可能原因：** RedirectURIs 配置错误

**解决方案：** 检查 RedirectURIs 是否正确配置为 SP 的回调地址
</Accordion>

<Accordion title="登录后返回错误">
**可能原因：** Issuer 或 Client ID 不正确

**解决方案：** 核对 SP SSO 设置中的 Issuer 和 Client ID 是否与专属集成平台中的一致
</Accordion>

<Accordion title="无法完成认证">
**可能原因：** 用户未被授权

**解决方案：** 检查用户是否在授权范围内
</Accordion>

<Accordion title="ID Token 验证失败">
**可能原因：** 签名算法不匹配

**解决方案：** 确认 ID Token 签名算法设置正确
</Accordion>

<Accordion title="单点登录成功但权限不足">
**可能原因：** 用户属性映射问题

**解决方案：** 检查 OIDC 声明（Claims）配置，确保必要的用户属性正确传递
</Accordion>
</Accordions>

## OIDC 与其他协议对比

### OIDC vs SAML

| 特性 | OIDC | SAML |
|------|------|------|
| 数据格式 | JSON/JWT | XML |
| 复杂度 | 相对简单 | 较为复杂 |
| 移动友好性 | 高 | 低 |
| 适用场景 | 现代 Web/移动应用 | 企业级传统应用 |

### OIDC vs CAS

| 特性 | OIDC | CAS |
|------|------|------|
| 标准化程度 | 国际标准 | 开源项目，主要教育领域 |
| 用户信息传递 | 丰富 | 有限 |
| 令牌格式 | JWT，自包含信息 | 不透明票据 |
| 生态系统 | 丰富多样 | 相对有限 |

## 常见问题（FAQ）

**Q1：OIDC 与 OAuth 2.0 有什么区别？**

A：OIDC 是建立在 OAuth 2.0 基础上的身份认证层。OAuth 2.0 主要解决授权问题，而 OIDC 解决身份认证问题。OIDC 在 OAuth 2.0 基础上添加了 ID Token、用户信息端点等功能。

**Q2：如何处理多个应用的单点登录？**

A：使用同一个身份提供商（如专属集成平台）为多个应用配置 OIDC，确保所有应用使用相同的 Issuer。用户登录一个应用后，访问其他应用时会自动重用现有会话。

**Q3：OIDC 如何保护用户隐私？**

A：OIDC 通过授权范围（Scope）和声明（Claims）控制应用可以访问的用户信息。用户在认证过程中可以看到应用请求的权限并选择同意或拒绝。

**Q4：如何实现单点登出？**

A：OIDC 支持前端通道和后端通道两种单点登出方式。配置登出回调地址后，用户从一个应用登出时，身份提供商可通知所有关联应用使其会话失效。
