1.
API规范
1.1
 响应格式
我们的API采用RESTful风格，并以JSON格式返回响应。除非另有说明，所有响应将遵循以下通用结构：
```json
{
    "result": true,          // 请求处理结果，成功为true，失败为false
    "error": "0",            // 错误码，成功为0，失败则为具体错误码
    "error_description": "SUCCESS",  // 详细的错误描述
    "data": {                // 具体的响应数据，详见各API文档
        // 数据内容
    }
}
```
1.2
 安全认证
在调用任何业务API之前，必须先通过“获取access_token”接口获取访问令牌（access_token）。后续请求中，需在Request Header中包含以下字段：
```xml
Authorization: Bearer <access_token>
```
其中<access_token>是您从“获取access_token”接口获得的令牌。
请确保在每次请求时都携带有效的access_token，以保证请求的安全性和有效性。

2.2
M2M 应用数据访问权限
除了上述 API 调用权限（如 scope 限定的操作类型）外，V5.6.2版本后M2M 应用对平台数据的访问和操作还受到严格的组织架构数据范围限制。
2.2.1
M2M 组织架构数据范围设置
在配置 M2M 应用时，您需要在管理控制台中明确选择 M2M 应用可访问的组织架构数据范围。此设置将直接限定 M2M 应用通过 API 可操作的用户、部门和角色所属的组织结构范围。
2.2.2
组织架构数据范围的含义
●
限定原则: M2M 应用的接口调用权限将严格受限于所配置的组织架构数据范围。它定义了 M2M 应用能够“看到”和“管理”的数据边界。
●
授权机制:
○
包含式允许: 如果 M2M 应用被授权访问某个父级部门（例如 "部门A"），则该应用将自动拥有操作 "部门A" 及其所有子部门下的用户、部门和角色的权限。这意味着数据范围是向下层级自动扩展的。
○
拒绝优先规则: 为了实现更精细的控制，您可以为特定子部门配置明确的“拒绝”规则。例如，即使 "部门A" 整体被授权，但如果您将 "部门A/子部门B" 列入拒绝范围，该 M2M 应用将无法操作 "部门A/子部门B" 及其所有子部门下的任何数据。任何冲突的“允许”规则都将被“拒绝”规则覆盖。
●
影响范围: 所有涉及用户管理、部门管理、角色管理等对组织架构有强依赖的 API 接口，都将受到此数据范围的限制。在调用这些 API 时，请确保请求中涉及的任何目标实体（例如：用户所属部门、目标部门、待操作的角色所属部门等）均在 M2M 应用的授权组织架构数据范围内且不被明确拒绝。
●
示例:
○
即使 M2M 应用拥有 user:create 的 API 操作权限，但如果尝试创建的用户其所属部门不在 M2M 应用被授权的组织架构数据范围内，该 user:create 操作仍将被系统拒绝。
○
类似地，若尝试查询或修改某个部门信息，但该部门不在 M2M 应用的授权范围内或被明确拒绝，相关 API 调用也将失败。

●

2.2.3
字段级访问控制
V5.8.1版本后增加了字段级配置，未配置的字段不允许访问，资源分为用户和部门。
2.2.3.1
资源类型
●
用户资源：涉及用户信息的字段访问控制
●
部门资源：涉及部门信息的字段访问控制
2.2.3.2
控制机制
●
只有明确配置的字段才允许访问
●
未配置的字段将被系统拒绝访问
●
提供更细粒度的数据安全控制
2.2.4
如何配置 M2M 组织架构数据范围和字段级访问控制
●
请登录您的集成平台管理控制台
●
导航至，应用中心 -> 应用管理 -> [选择您的应用] -> [授权范围] -> [M2M授权]
●
在该界面，您可以选择或勾选 M2M 应用可以访问的组织部门范围，并根据需要配置具体的允许或拒绝规则
●
根据需要配置具体的允许或拒绝规则
●
设置字段级访问权限配置。

attrs参数：允许客户端自定义需要返回的用户属性字段，这有助于减少不必要的数据传输，提高接口响应速度。支持的属性包括但不限于用户的唯一标识符（sub）、姓名、组织ID列表、电子邮件地址等。请注意，尽管可以请求多个属性，但并非所有应用都拥有访问某些敏感信息（如手机号码）的权限，因此在实际响应中可能不会包含这些字段。
●
分页功能：通过设置page和size参数来控制返回的数据量及当前查看的位置。这对于处理大量用户数据尤为重要，因为它可以帮助前端应用程序更高效地展示信息，并改善用户体验。同时，请确保合理选择每页显示的记录数，以避免一次性加载过多数据导致性能下降。
●
模糊搜索：利用q参数可以根据用户名、真实姓名、手机号或邮箱地址进行模糊匹配查询。这项功能对于快速定位特定用户非常有用，尤其是在大型企业环境中。然而，在执行此类搜索时也应注意保护个人隐私，避免未经授权的信息泄露。
●
身份验证：每次调用本接口都需要附带有效的access_token作为认证凭据。获取此令牌的过程应遵循OAuth 2.0标准流程，并且建议定期刷新以保证安全性。此外，不同角色的账号可能会有不同的访问级别，因此在设计相关业务逻辑时需考虑这一点。
●
错误处理：如果请求过程中出现任何问题，服务器将以JSON格式返回详细的错误信息，包括错误代码(error)和描述(error_description)。开发者应当根据这些反馈采取适当措施，比如重新尝试请求、提示用户检查输入等，从而提升系统的健壮性和容错能力。
●
性能考量：虽然此API提供了强大的查询功能，但在实际部署时仍需关注其对后端数据库的压力。特别是在高并发场景下，频繁的大范围查询可能导致服务响应变慢甚至超时。为此，建议实施缓存策略、优化索引结构以及限制单次请求的最大返回条目数量等方式来缓解潜在性能瓶颈。
