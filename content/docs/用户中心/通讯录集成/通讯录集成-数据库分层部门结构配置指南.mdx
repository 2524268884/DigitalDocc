## 概述
本指南详细说明如何在集成平台中配置数据库连接器，特别关注实现多级部门层次结构的同步。通过数据库集成，您可以将数据库中维护的组织架构和用户数据同步至平台，进一步整合企业身份数据，确保各应用系统的身份信息一致性。
数据库集成特别适用于以下场景：
- 企业已有基于数据库的用户管理系统
- 需要从多个数据源整合用户信息
- 组织结构以分层部门结构（如一级部门/二级部门/三级部门）形式存储
- 需要灵活定制用户和部门属性的同步规则
1. 部门路径结构说明
在路径结构集成模式下，系统采用部门追加方式进行同步。这是一种特殊的同步机制，您需要了解以下特性：
重要说明：
- 当用户部门发生变更时，原有部门结构仍会保留在集成平台中
- 例如：张三的部门原先是 A/B/C，后来修改为 A/B/D，同步后 C 部门仍会保留在集成平台中
- 这种机制确保不会因个别用户部门变更而影响其他用户所属的组织结构
路径结构优势：
- 完整保留多级部门层次关系
- 简化部门关系的维护
- 支持复杂的组织架构表达
- 便于用户在多部门间移动时的数据处理

### 配置SQL脚本
在数据库执行脚本部分，需要分别配置用户数据和部门数据的查询SQL：
## 用户数据查询SQL：

```sql
SELECT
  user_id as id,
  username,
  real_name as name,
  department_path as path,
  employee_code as usercode,
  position_name as position
FROM employee_table;
```

## 部门数据查询SQL（可选，用于同步部门额外属性）：

```sql
SELECT
  dept_id as id,
  parent_dept_id as parentid,
  dept_name as name,
  dept_type as type
FROM department_table;
```

### 验证同步结果
1. 同步完成后，导航至「用户中心 > 用户管理」查看导入的用户数据
2. 检查组织结构，确认多级部门层次是否正确创建
- 验证用户是否关联到了正确的部门位置
1. 配置注意事项
2. 路径字段格式：
  - 确保用户查询结果中包含path字段，该字段值应为完整的部门层级路径
  - 路径格式示例："总公司/人力资源部/招聘组"
  - 路径分隔符通常为"/"，确保数据库中存储的路径使用统一分隔符
1. 部门追加机制：
  - 路径结构使用部门追加模式，原有部门不会因用户部门变更而被删除
  - 这可能导致组织结构中存在一些"孤岛"部门（没有用户的部门）
  - 如需清理无用部门，需手动操作或编写专门的清理脚本
1. 数据一致性：
  - 确保数据库中部门路径的格式一致性
  - 避免同一级别部门使用重复名称，可能导致部门合并或识别错误
  - 确保路径层级不要过深（建议不超过6级），以免影响性能和可维护性
1. 账号关联规则：
  - 正确设置账号关联规则，以准确识别用户身份
  - 常用的关联字段包括：工号、用户名、邮箱等唯一标识
  - 配置示例：数据库用户表.工号 = 集成平台.用户编号
1. 常见问题与故障排除
### 同步问题
## 部门层级不正确
  - 原因：path字段格式不符合要求或分隔符不一致
  - 解决方法：
■
检查SQL查询返回的path字段值
■
确保使用统一的分隔符（通常是"/"）
■
验证路径不包含特殊字符或多余空格
1. 部门名称显示乱码
  - 原因：字符编码不匹配
  - 解决方法：
■
检查数据库连接字符集设置
■
确保SQL查询结果使用UTF-8编码
■
必要时在SQL中使用转换函数处理编码
1. 重复部门问题
  - 原因：路径表达不一致导致相同部门被创建多次
  - 解决方法：
■
标准化数据库中的部门路径表达
■
检查是否存在空格、大小写或特殊字符差异
■
考虑使用部门ID而非名称作为唯一标识
### 性能问题
## 同步速度慢
  - 原因：SQL查询效率低或返回数据量过大
  - 解决方法：
■
优化SQL查询，添加必要的索引
■
减少不必要的字段查询
■
考虑分批同步或增量同步
1. 数据库负载高
  - 原因：同步过程中数据库查询资源消耗大
  - 解决方法：
■
在非业务高峰期执行同步
■
优化SQL，避免全表扫描
■
考虑从从库或备份库读取数据
### 常见问题解答 (FAQ)
1. Q: 如何处理用户属于多个部门的情况？A: 有两种方法：
  - 在path字段中使用分号分隔多个部门路径："路径1;路径2"
  - 为同一用户创建多条记录，每条记录对应一个部门路径
2. Q: 数据库架构变更后，如何更新集成配置？A: 修改SQL查询语句适应新架构，并确保字段映射关系正确。变更后先在测试环境验证，确认无误后再应用到生产环境。
3. Q: 如何处理历史数据迁移问题？A: 建议先执行一次全量同步建立基准，之后再配置增量同步。对于大量历史数据，可以考虑分批导入，每批控制在合理的数量范围内。
4. Q: 可以同时从多个数据库表获取用户信息吗？A: 可以，通过在SQL中使用JOIN语句关联多个表。确保查询结果包含所有必要字段，特别是用户标识和部门路径信息。

### 自定义SQL查询优化
对于复杂的组织结构，可以使用更高级的SQL查询：
1. 处理多部门关系：`sqlSELECTu.user_id as id,u.username,u.real_name as name,LISTAGG(d.deptpath, ';') WITHIN GROUP (ORDER BY d.primaryflag DESC) as path,u.employee_code as usercode,u.position_name as positionFROM users uJOIN userdepartment ud ON u.userid = ud.user_idJOIN departments d ON ud.deptid = d.deptidGROUP BY u.userid, u.username, u.realname, u.employeecode, u.positionname;`
2. 包含部门特性的查询：`sqlSELECTdept_id as id,parentdeptid as parentid,dept_name as name,dept_manager as manager,dept_code as code,dept_phone as phone,dept_address as address,statusFROM departmentsWHERE status = 'ACTIVE';`
### 增量同步配置
对于大型数据库，推荐使用增量同步提高效率：

```sql
SELECT
  user_id as id,
  username,
  real_name as name,
  department_path as path,
  employee_code as usercode,
  position_name as position,
  last_update_time
FROM employee_table
WHERE last_update_time > :last_sync_time;
```
其中:last_sync_time是系统变量，会自动替换为上次同步的时间戳。
