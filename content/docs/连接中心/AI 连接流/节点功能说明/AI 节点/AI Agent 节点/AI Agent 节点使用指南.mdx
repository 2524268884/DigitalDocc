本文档是 AI Agent 节点的综合性使用指南，涵盖节点架构、核心概念、配置方法及最佳实践。

## 什么是 AI Agent 节点

AI Agent 节点是 AI 连接流中的核心智能组件，能够接收用户输入、调用大语言模型进行推理、并通过工具节点执行实际的业务操作。与简单的模型调用不同，AI Agent 具备**自主决策**能力，可以根据上下文判断何时调用何种工具。

## 节点架构

AI Agent 节点由三个核心子节点组成，缺一不可：

| 子节点 | 作用 | 文档 |
|--------|------|------|
| 模型子节点 | 负责语言理解与生成 | [模型子节点](./模型子节点.mdx) |
| 存储子节点 | 管理对话上下文与记忆 | [存储子节点](./存储子节点.mdx) |
| 工具子节点 | 提供业务操作能力 | [MCP SERVER 工具](./工具子节点/MCP%20SERVER%20工具.mdx) |

```
┌─────────────────────────────────────────┐
│              AI Agent 节点               │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │ 模型节点 │  │ 存储节点 │  │ 工具节点 │ │
│  │ (model) │  │(memory) │  │ (tools) │ │
│  └────┬────┘  └────┬────┘  └────┬────┘ │
│       │            │            │       │
│       └────────────┼────────────┘       │
│                    ▼                    │
│            ┌───────────────┐             │
│            │   大语言模型   │             │
│            └───────────────┘             │
└─────────────────────────────────────────┘
```

## 核心概念

### 系统提示词 (System Prompt)

系统提示词定义了 AI Agent 的角色定位和行为规则。优秀的提示词应当：

- 明确角色身份（如"你是一个客服助手"）
- 清晰说明能力边界
- 规范输出格式
- 包含处理异常情况的指引

### 用户提示词 (User Prompt)

用户提示词是用户输入的模板，支持变量绑定：

```
请查询客户 {{customer_id}} 的订单状态，
订单编号为 {{order_id}}。
```

### 对话上下文

通过存储子节点，AI Agent 能够在多轮对话中保持上下文一致。每次对话时，存储节点会自动加载历史消息，使模型理解对话背景。

## 子节点详解

### 模型子节点

模型子节点负责与大语言模型通信，主要配置包括：

| 参数 | 说明 | 建议值 |
|------|------|--------|
| 模型服务商 | 通义千问 / DeepSeek / OpenAI / Claude | 根据业务需求 |
| 温度 (temperature) | 控制随机性，值越高越有创意 | 0.7 |
| 最大 Token | 生成内容最大长度 | 1024 |
| 超时时间 | 请求最大等待时长（秒） | 60 |

详细配置请参阅 [模型子节点](./模型子节点.mdx)。

### 存储子节点

存储子节点管理对话历史和上下文，核心参数：

| 参数 | 说明 |
|------|------|
| 消息 Key | 标识对话的唯一ID，建议使用 user_id 或 session_id |
| 上下文窗口长度 | 控制引用历史消息数量，太长增加成本，太短丢失上下文 |
| 存储类型 | 短期（会话内）或长期（持久化） |

详细配置请参阅 [存储子节点](./存储子节点.mdx)。

### 工具子节点

工具子节点让 AI Agent 能够调用业务系统。通过 MCP (Model Context Protocol) 协议，Agent 可以：

- 查询业务数据
- 创建/更新 records
- 执行审批流程
- 调用第三方 API

详细配置请参阅 [MCP SERVER 工具](./工具子节点/MCP%20SERVER%20工具.mdx)。

## 典型使用场景

### 场景一：智能客服

用户通过对话咨询问题，AI Agent 自动识别意图并调用工具查询答案。

```
用户 → 对话触发 → AI Agent → 查询订单工具 → 回复用户
```

### 场景二：业务代办

用户说"帮我创建一条待办"，AI Agent 自动解析参数并调用待办系统 API。

### 场景三：数据查询助手

用户询问业务数据，AI Agent 组合使用 SQL 节点和模型节点返回分析结果。

## 最佳实践

### 提示词设计

1. **角色明确**：开头即说明 Agent 身份
2. **格式规范**：指定输出格式（如 JSON、表格）
3. **边界清晰**：说明什么能做、什么不能做
4. **容错处理**：预设异常情况处理方式

示例：

```
你是一个企业订单客服助手。
职责：
- 查询订单状态和物流信息
- 解答常见问题
- 无法解答时转人工

输出格式：
- 正常回复：直接回答
- 无法解答：回复"抱歉，我无法回答这个问题，已转人工处理"
```

### 工具配置

1. **动作描述要业务化**：避免技术术语，用业务语言描述
2. **参数尽可能 AI 生成**：减少用户输入负担
3. **同类工具放同一节点**：便于管理和维护
4. **避免工具堆积**：每个 MCP 节点建议不超过 10 个动作

### 上下文管理

1. **消息 Key 要有业务含义**：使用 user_id 而不是临时 ID
2. **窗口长度合理设置**：一般 5-10 条消息足够
3. **敏感数据不进入上下文**：在工具层处理脱敏

## 调试与排错

### 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| AI 回复不符合预期 | 提示词不够明确 | 优化系统提示词，增加示例 |
| 工具未被调用 | 提示词中未说明可用工具 | 在提示词中列出工具及用途 |
| 上下文丢失 | 消息 Key 不一致 | 检查 Key 生成逻辑 |
| 请求超时 | 模型响应慢或网络问题 | 增大超时时间或切换模型 |
| 工具调用失败 | 参数格式错误或权限不足 | 检查工具配置和参数传递 |

### 调试技巧

1. **先单独测试子节点**：确认模型、存储、工具各自正常
2. **查看节点运行日志**：定位问题发生在哪个环节
3. **逐步增加复杂度**：从简单对话开始，逐步添加工具
4. **使用变量监控**：观察参数传递是否正确
