## 概述

应用授权管理用于控制哪些用户可以访问应用，以及应用间 API 的调用权限。平台支持两大类授权：

- **用户访问授权**：定义哪些用户可以访问应用，支持全部授权、组织架构授权、角色授权、用户授权
- **M2M 接口授权**：管理应用间 API 调用的权限范围，适用于系统集成场景

## 前提条件

- 拥有专属集成平台管理员账号
- 已完成应用的基础创建与配置

## 用户访问授权

### 授权模式说明

| 模式 | 适用场景 |
|------|---------|
| **全部授权** | 允许所有平台用户访问，适用于企业内通用系统（OA、邮箱等） |
| **组织架构授权** | 基于部门组织结构授权，适用于部门专属应用 |
| **角色授权** | 基于静态或动态角色授权，适用于按职能控制访问 |
| **用户授权** | 指定特定用户，适用于仅少数人员使用的应用 |

<Callout type="info">
多种授权模式可同时配置，各模式间采用**逻辑"或"**关系——满足任一条件即可访问。授权变更**实时生效**，无需用户重新登录。
</Callout>

### 配置授权范围

<Steps>
<Step>
### 步骤 1：进入应用编辑页

进入**应用中心 > 应用管理**，选择目标应用，点击**编辑**。
</Step>

<Step>
### 步骤 2：切换到授权范围选项卡

点击**授权范围**选项卡，选择授权类型。
</Step>

<Step>
### 步骤 3：配置具体授权内容

<Tabs items={['组织架构授权', '角色授权', '用户授权']}>
<Tab value="组织架构授权">
1. 点击**选择部门**
2. 在组织树中勾选需要授权的部门
   - 选择父部门会同时授权所有子部门的用户
   - 仅授权子部门可单独勾选该子部门
3. 点击**确定**完成选择

<Callout type="info">
组织架构授权支持继承机制：子部门用户自动获得父部门的授权。
</Callout>
</Tab>

<Tab value="角色授权">
1. 点击**选择角色**
2. 在角色列表中勾选需要授权的角色：
   - **静态角色**：手动分配给用户的固定角色
   - **动态角色**：基于用户属性自动计算的角色
3. 点击**确定**完成选择
</Tab>

<Tab value="用户授权">
1. 点击**添加用户**
2. 通过以下方式选择用户：
   - 按用户名称搜索
   - 按用户属性筛选
   - 从组织架构中选择
   - 从角色中选择
3. 勾选目标用户，点击**确定**完成选择
</Tab>
</Tabs>
</Step>

<Step>
### 步骤 4：保存配置

点击**保存**，授权配置立即生效。
</Step>
</Steps>

## M2M 接口授权

M2M（Machine-to-Machine）接口授权用于管理应用间 API 调用的权限范围，确保应用仅能调用被授权的 API。

### 适用场景

- 第三方应用调用平台 API
- 企业自建应用间的 API 调用
- 自动化集成场景下的系统间通信

### 可授权的 API 权限分类

| 分类 | 包含操作 |
|------|---------|
| **用户管理** | 查询用户、创建用户、编辑用户、调整部门、删除用户 |
| **部门管理** | 查询部门、创建部门、编辑部门、删除部门 |
| **角色管理** | 查询角色、创建角色、编辑角色、删除角色、角色成员管理 |

<Callout type="warn">
对于创建、编辑、删除类操作，系统会显示安全风险提示。涉及数据写入或删除的操作请严格遵循最小权限原则，谨慎授权。
</Callout>

### 配置 M2M 授权

<Steps>
<Step>
### 步骤 1：进入应用配置

在**应用中心 > 应用管理**，选择目标应用，点击**编辑 > 权限范围**选项卡。
</Step>

<Step>
### 步骤 2：开启 M2M 授权

找到 **M2M 授权**区域，开启 **M2M 应用授权设置**开关。
</Step>

<Step>
### 步骤 3：选择 API 权限

勾选该应用所需的 API 权限分类及具体操作项，点击**保存**完成配置。
</Step>
</Steps>

### API 调用示例

配置 M2M 授权后，应用通过以下方式获取 Access Token 并调用 API：

**获取 Access Token（客户端凭证模式）：**

```http
POST /oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id={CLIENT_ID}
&client_secret={CLIENT_SECRET}
```

**使用 Access Token 调用 API：**

```http
GET /api/v1/users
Authorization: Bearer {ACCESS_TOKEN}
```

## 授权审计

平台提供完整的授权变更审计功能：

- **操作记录**：记录所有授权变更操作
- **变更详情**：记录变更前后的授权状态
- **操作人员**：记录执行变更的管理员信息
- **时间戳**：记录变更的精确时间

## 最佳实践

**用户访问授权**
- 优先使用**角色和组织架构授权**，相比直接用户授权更易于批量管理
- 合理规划角色体系，避免角色数量过多
- 定期清理不再需要的授权，遵循最小权限原则

**M2M 接口授权**
- 仅授予应用必要的 API 权限
- 对创建、修改、删除操作进行严格控制
- 定期轮换 Client Secret（建议每 3-6 个月一次）
- 监控 API 调用情况，发现异常及时处理

## 常见问题

<Accordions>
<Accordion title="用户无法访问已授权应用">
1. 检查用户是否属于已授权的组织或角色
2. 确认用户账号处于活跃状态（未被停用）
3. 验证应用本身是否处于启用状态
4. 检查多种授权模式是否有冲突配置
</Accordion>

<Accordion title="M2M API 调用失败">
1. 检查 Access Token 是否有效（可能已过期）
2. 确认应用已开启对应 API 的 M2M 授权
3. 验证 API 请求格式（请求头、参数）是否正确
4. 检查 Client ID 和 Client Secret 是否正确
</Accordion>

<Accordion title="授权变更后未生效">
1. 确认授权更改已保存（检查页面是否有未保存提示）
2. 清理用户浏览器缓存和 Cookie
3. 如用户已有活跃会话，可能需要重新登录后生效
</Accordion>

<Accordion title="如何批量授权大量用户">
- 使用**组织架构授权**替代大量的用户直接授权
- 将用户加入角色组，通过**角色授权**批量覆盖
- 利用用户标签/属性功能进行动态分组授权（ABAC）
</Accordion>
</Accordions>
