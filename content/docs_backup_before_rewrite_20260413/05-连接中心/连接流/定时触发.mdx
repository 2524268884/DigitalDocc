## 概述

定时触发器基于时间调度执行AI FLOW流程，支持灵活的定时任务配置。通过标准的Cron表达式，可以实现分钟级到年级的各种定时执行需求，适用于定期数据处理、报表生成、系统维护等场景。

## 功能特性
- **Cron表达式支持**：支持标准的Quartz Cron语法
- **时间范围控制**：可设置开始和结束日期限制执行周期
- **自动调度**：集成Quartz调度框架，稳定可靠
- **状态管理**：支持启用/禁用定时任务
- **执行记录**：记录每次执行的历史和结果

## 配置参数

### startDate
- **类型**：String
- **描述**：定时任务的开始日期
- **格式**：YYYY-MM-DD
- **示例**：`2024-01-01`
- **说明**：任务只会在此日期之后开始执行

### endDate
- **类型**：String
- **描述**：定时任务的结束日期
- **格式**：YYYY-MM-DD
- **示例**：`2024-12-31`
- **说明**：任务在此日期之后停止执行

### cycle
- **类型**：String
- **描述**：Cron表达式定义执行周期
- **格式**：标准的Cron表达式（秒 分 时 日 月 周 年）
- **示例**：
    - `0 0/30 * * * ?`：每30分钟执行一次
    - `0 0 9 * * ?`：每天早上9点执行
    - `0 0 0 1 * ?`：每月1号执行

## Cron表达式语法

Cron表达式由7个字段组成，用空格分隔：
```
秒 分 时 日 月 周 年
```

### 字段说明
- **秒**：0-59
- **分**：0-59
- **时**：0-23
- **日**：1-31
- **月**：1-12或JAN-DEC
- **周**：1-7或SUN-SAT（1=SUN）
- **年**：可选，1970-2099

### 特殊字符
- `*`：匹配所有值
- `?`：不指定值（用于日和周冲突时）
- `-`：范围，如`10-12`
- `,`：列表，如`1,3,5`
- `/`：增量，如`0/15`（每15单位）
- `L`：最后，如`L`（每月最后一天）
- `W`：工作日，如`15W`（每月15号的最近工作日）
- `#`：第几个，如`2#1`（每月第一个星期一）

## 常见Cron表达式

| 表达式 | 含义 |
|---------|------|
| `0 0/5 &#42; &#42; &#42; ?` | 每5分钟执行一次 |
| `0 0 0 &#42; &#42; ?` | 每天午夜执行 |
| `0 30 9 &#42; &#42; MON-FRI` | 工作日早上9:30执行 |
| `0 0 0 1 1 ?` | 每年1月1日执行 |
| `0 0/30 9-17 &#42; &#42; MON-FRI` | 工作日9点到17点每30分钟执行 |
| `0 0 0 L &#42; ?` | 每月最后一天执行 |
| `0 15 10 ? &#42; 6L` | 每月最后一个星期五上午10:15执行 |

## 使用方法

### 1\. 配置Cron触发器

在StartEvent节点中选择Cron作为触发器类型。

### 2\. 设置时间参数
```
{
  "startDate": "2024-01-01",
  "endDate": "2024-12-31", 
  "cycle": "0 0 9 * * ?"
}
```

### 3\. 验证表达式

使用在线Cron表达式验证工具确认语法正确性。

### 4\. 启用任务

配置完成后，定时任务会自动启用并开始调度。

## 执行逻辑

### 调度流程
1. 解析Cron表达式
2. 计算下次执行时间
3. 等待到达执行时间
4. 检查是否在有效日期范围内
5. 触发流程执行
6. 记录执行结果
7. 计算并等待下次执行时间

### 错过执行处理

如果系统宕机导致错过执行时间点：
- 系统重启后会检查是否有错过的执行
- 根据配置决定是否补执行
- 默认跳过错过的执行，继续下次正常调度

## 管理功能

### 任务状态控制
- **启用/禁用**：可临时暂停或恢复定时任务
- **立即执行**：手动触发一次执行（不影响正常调度）
- **修改配置**：支持运行时修改Cron表达式等参数

### 执行历史
- 记录每次执行的时间戳
- 执行结果（成功/失败）
- 执行耗时
- 错误信息（失败时）

## 最佳实践

### 1\. 合理设置执行频率
- 避免过于频繁的执行（如每分钟）
- 根据业务需求平衡实时性和系统负载
- 考虑下游系统处理能力

### 2\. 设置适当的时间范围
- 为临时任务设置明确的结束日期
- 长期任务可设置较远的结束日期或不设置

### 3\. 使用测试环境验证
- 在测试环境验证Cron表达式
- 检查时区设置是否正确
- 确认系统时间同步

### 4\. 监控和告警
- 设置执行失败告警
- 监控执行成功率
- 定期检查执行历史

## 故障排除

### 任务未执行
1. 检查Cron表达式语法
2. 验证开始/结束日期设置
3. 确认任务状态为启用
4. 查看系统日志是否有错误

### 执行时间不准确
1. 检查系统时区设置
2. 确认服务器时间同步
3. 查看是否有时钟漂移

### 重复执行
1. 检查Cron表达式是否过于宽泛
2. 验证是否有多个相同配置的任务
3. 查看任务执行历史确认重复原因

## 性能考虑

### 系统负载
- 高频任务会增加系统负载
- 建议将密集任务分散到不同时间点
- 监控CPU和内存使用情况

### 并发控制
- 单次执行未完成时，后续执行会被跳过
- 避免长时间运行的任务影响调度

## 常见问题

### Q: Cron表达式如何测试？

**A:** 可以使用在线工具如crontab.guru或cron表达式验证器来测试和验证表达式。输入表达式后可以看到下次执行时间。

### Q: 支持秒级精度吗？

**A:** 是的，Cron表达式包含秒字段，支持秒级定时精度。但实际执行可能受系统调度器影响，有轻微延迟。

### Q: 如何处理夏令时变化？

**A:** 系统使用UTC时间存储，执行时根据服务器时区转换。建议在夏令时变化期间监控任务执行情况。

### Q: 最多可以配置多少个定时任务？

**A:** 理论上没有限制，但建议单个系统不超过1000个活跃定时任务。过多任务会影响调度性能。

### Q: 任务执行失败后会如何处理？

**A:** 失败的任务不会影响后续调度，会继续按计划执行。失败信息会记录在执行历史中，支持配置告警通知。
