## 1\. 概述

专属集成平台的身份自动化集成功能允许企业通过REST API方式与任意外部身份系统建立连接，实现灵活的身份数据同步。本指南详细说明如何配置REST API身份自动化集成，通过自定义流程编排方式，逐步从外部系统获取用户信息并写入专属集成平台。

### 集成优势
- **通用性强**：可连接任何提供REST API的身份系统或应用
- **高度可定制**：通过流程编排自定义数据获取和处理逻辑
- **灵活适配**：适应各类API接口格式和认证方式
- **端到端集成**：支持完整的用户、部门和角色数据同步
- **简化开发**：无需编写代码，通过可视化配置实现集成

### 适用场景
- **自建系统对接**：连接企业内部自建的HR系统或业务系统
- **SaaS应用集成**：同步来自SaaS应用的用户和组织数据
- **混合数据源整合**：组合多个REST API数据源的用户信息
- **特殊格式处理**：需要对API返回数据进行复杂转换的场景
- **定制化同步逻辑**：需要特定业务规则处理的身份同步场景

## 2\. 前置准备

### 必要条件
- 专属集成平台管理员账号，具备通讯录集成配置权限
- 目标系统提供的REST API接口文档，包含：
    - API接口地址和请求方法
    - 认证方式（如API Key、OAuth等）
    - 请求参数格式和必填项
    - 响应数据结构说明
- 了解数据映射关系，包括：
    - 源系统中用户/部门字段与目标字段的对应关系
    - 特殊字段的处理逻辑（如多值字段、状态转换等）
- 确认网络环境允许专属集成平台访问目标API（如防火墙配置、IP白名单等）

### API接口准备清单

在配置前，请确保您已获取并整理以下API信息：

| API类型 | 接口地址 | 请求方法 | 认证方式 | 功能说明 | 是否必需 |
|---------|------------|------------|------------|------------|------------|
| 认证接口 | 例：/api/token | POST | Basic Auth | 获取访问令牌 | 视API而定 |
| 部门列表接口 | 例：/api/departments | GET | Bearer Token | 获取所有部门 | 是 |
| 部门详情接口 | 例：/api/departments/\{id\} | GET | Bearer Token | 获取单个部门详情 | 可选 |
| 用户列表接口 | 例：/api/users | GET | Bearer Token | 获取所有用户 | 是 |
| 用户详情接口 | 例：/api/users/\{id\} | GET | Bearer Token | 获取单个用户详情 | 可选 |
| 角色列表接口 | 例：/api/roles | GET | Bearer Token | 获取所有角色 | 可选 |

请根据您的实际API情况填写上表，并准备好接口的请求参数和响应数据示例，这将有助于后续的集成配置。

- **与选择节点合并**：同步的组织架构将合并到选定节点，保持原节点ID

**最佳实践：**
- 确保上游字段名与API响应数据结构完全匹配，包括大小写
- 对于多值字段（如用户可能属于多个部门），确认字段是否为数组类型
- 若API返回的字段名称不一致，可在流程编排阶段进行转换

### 3.3 配置用户信息属性映射字段

填好并保存成功基础信息后，需要配置相应的用户信息属性映射字段：

![image.png](/images/身份自动化集成-REST API配置/0e6ac1e1-7090-4b4e-b3e0-3c082d26ca88.png)

**属性映射方式：**
- **不映射**：不会同步用户的该属性至专属集成平台
- **仅创建**：只有在创建用户的时候会同步该属性，更新用户信息时不会同步
- **创建和更新**：创建和更新用户信息时均同步该属性

**映射颜色标识含义：**
- **紫色**：表示必填字段，必须配置映射关系
- **绿色**：表示已配置映射的可选字段
- **灰色**：表示未配置映射的可选字段

### 3.4 配置身份自动化集成连接器的同步策略

配置好用户信息属性映射内容后，还要配置集成的同步策略，如下图：

![image.png](/images/身份自动化集成-REST API配置/e7260ec6-162c-4c8d-9abb-04b676efcc6d.png)

**同步策略字段说明：**

| 参数名称 | 说明 | 可选值 | 建议配置 | 适用场景 |
|------------|------|---------|------------|------------|
| 同步方式 | 数据同步的触发方式和范围 | 自动增量/手动全量/自动全量 | 自动增量同步 | 日常实时同步 |
| 同步周期 | 自动同步的时间间隔 | 5分钟到24小时多个选项 | 每30分钟 | 平衡实时性和系统负载 |
| 开始时间 | 首次自动同步的启动时间 | 自定义时间 | 当前时间 | 立即开始同步 |
| 账号关联 | 用户ID关联映射规则 | 多字段组合 | 根据企业实际情况 | 确保用户唯一性 |

**同步方式详细说明：**
- **自动增量同步**：系统按固定频率同步新增和修改的用户，适合日常维护
- **手动全量同步**：管理员手动触发同步全量的用户（包括删除的用户），适合初始化或大规模变更
- **自动全量同步**：系统按固定频率同步全量的用户（包括删除的用户），适合定期数据校验

**同步周期选项：** 系统支持"每5分钟"、"每10分钟"、"每30分钟"、"每小时"、"每2小时"、"每6小时"、"每12小时"、"24小时"多种集成周期选项，可根据企业数据变更频率和同步实时性需求选择。

**账号关联设置技巧：**
- 优先使用稳定唯一的属性作为关联依据（如工号、唯一ID等）
- 可以设置多条关联规则，形成"或"关系，提高匹配成功率
- 针对特殊情况，可定制复杂的匹配规则

**待删除用户阈值设置：**
- **小型企业(\2000用户)**：建议3-5%，提供足够的误操作保护

**流程编排关键节点配置：**

#### HTTP请求节点配置

HTTP请求节点是与外部API交互的核心组件，正确配置此节点对于成功获取数据至关重要：

| 配置项 | 说明 | 配置示例 | 注意事项 |
|---------|------|------------|------------|
| 请求方法 | HTTP方法类型 | GET/POST/PUT | 根据API文档选择 |
| URL | 接口地址 | https://api.example.com/users | 可使用变量构建动态URL |
| 查询参数 | URL参数 | page=1&size=100 | 分页、筛选等参数 |
| 请求头 | HTTP请求头 | Content-Type: application/json | 认证信息、内容类型等 |
| 请求体 | POST/PUT请求数据 | \{"filter": \{"active": true\}\} | JSON格式数据 |
| 超时设置 | 请求超时时间 | 30000 (30秒) | 根据API响应时间设置 |
| 错误处理 | 请求失败处理机制 | 重试、跳过或终止 | 影响流程稳定性 |

**最佳实践：**
- 使用变量存储认证令牌，在多个请求间共享
- 为请求添加适当的错误处理和重试逻辑
- 使用日志节点记录请求和响应，便于调试
- 考虑API限流，添加适当的请求间隔

#### 数据转换节点配置

数据转换节点用于处理API返回的数据，将其映射和转换为平台所需的格式：

| 功能 | 使用场景 | 配置示例 | 注意事项 |
|------|------------|------------|------------|
| 字段映射 | 字段名称不一致 | response.userName → user.name | 确保类型匹配 |
| 数据过滤 | 筛选特定数据 | users.filter(u =\> u.status === 'active') | 注意空值处理 |
| 数据合并 | 组合多个API结果 | \{...basicInfo, ...detailInfo\} | 解决字段冲突 |
| 格式转换 | 日期格式统一 | formatDate(user.createTime, 'YYYY-MM-DD') | 确保格式规范 |
| 默认值设置 | 处理确实字段 | user.title \|\| '无职位' | 保证数据完整性 |
| 条件处理 | 状态值转换 | user.status === 1 ? 'active' : 'inactive' | 逻辑清晰明确 |

\| 默认值设置 \| 处理缺失字段 \| user.title \|\| '无职位' \| 保证数据完整性 \|
\| 条件处理 \| 状态值转换 \| user.status === 1 ? 'active' : 'inactive' \| 逻辑清晰明确 \|

**最佳实践：**
- 创建中间变量存储转换过程的临时数据
- 使用注释明确转换逻辑，便于后期维护
- 对重复使用的转换逻辑创建公共函数
- 特别注意数组类型数据的处理

#### 循环处理节点配置

循环节点用于处理集合数据，如遍历部门列表、用户列表等：

| 配置项 | 说明 | 配置示例 | 注意事项 |
|---------|------|------------|------------|
| 集合来源 | 要遍历的数据集合 | response.departments | 确保是数组类型 |
| 迭代变量 | 当前循环项的变量名 | department | 描述性命名 |
| 索引变量 | 当前循环索引变量名 | index | 可用于跟踪进度 |
| 并行设置 | 是否并行处理 | 串行/并行 | 考虑API限制和数据依赖 |
| 出错处理 | 单项处理失败的策略 | 继续/中断/重试 | 影响整体流程稳定性 |

**最佳实践：**
- 对大量数据实施分批处理，避免单次处理过多项目
- 添加进度日志，便于监控长时间运行的循环
- 考虑添加条件判断，跳过不需要处理的项目
- 设置合理的错误处理策略，避免单项失败影响整体

## 4\. 典型流程编排示例

### 4.1 基础REST API同步流程

下面是一个基本的REST API身份同步流程示例，展示了如何获取和处理用户与部门数据：
```
开始
 ├── 获取认证令牌 (HTTP请求 - POST)
 │   └── 保存令牌到变量
 ├── 获取部门列表 (HTTP请求 - GET)
 │   ├── 数据转换 (处理部门数据格式)
 │   └── 循环处理部门
 │       ├── 部门数据映射
 │       └── 保存部门信息
 ├── 获取用户列表 (HTTP请求 - GET)
 │   ├── 数据转换 (处理用户数据格式)
 │   └── 循环处理用户
 │       ├── 获取用户详情 (HTTP请求 - GET)
 │       ├── 用户数据映射和转换
 │       └── 保存用户信息
 └── 结束
```

### 4.2 分页处理流程示例

处理大量数据时，API通常会使用分页机制，以下是处理分页的流程示例：
```
开始
 ├── 获取认证令牌 (HTTP请求)
 ├── 初始化分页变量 (page=1, hasMore=true)
 ├── 循环处理 (当hasMore为true)
 │   ├── 获取用户列表 (带分页参数)
 │   ├── 数据处理和保存
 │   ├── 检查是否有下一页
 │   └── 更新分页变量
 └── 结束
```

### 4.3 错误处理与重试机制

健壮的集成流程需要妥善处理可能的错误和异常情况：
```
开始
 ├── 获取令牌 (带重试逻辑)
 │   └── 错误处理：重试3次，间隔递增
 ├── 获取数据 (带错误捕获)
 │   ├── 尝试获取数据
 │   ├── 错误处理分支
 │   │   ├── 401/403错误：重新获取令牌后重试
 │   │   ├── 429错误：等待后重试 (限流处理)
 │   │   ├── 5xx错误：等待后重试
 │   │   └── 其他错误：记录日志，继续流程
 │   └── 数据处理
 └── 结束
```

## 5\. 测试与验证

### 5.1 连接测试

完成流程编排后，应进行全面测试以确保配置的正确性：
1. **单节点测试**：从流程编排界面，选择关键HTTP节点，点击"测试"按钮验证单个API调用
2. **参数校验**：确认所有请求参数、头信息和认证信息正确配置
3. **响应分析**：检查API响应数据是否符合预期，格式是否正确
4. **错误模拟**：故意使用错误参数，测试错误处理机制是否有效
5. **完整流程测试**：执行整个流程，观察数据处理和传递情况

**测试结果评估：**
- 检查每个API调用的状态码和响应时间
- 验证数据转换和映射结果是否符合预期
- 确认错误处理和重试机制正常工作

### 10.5 调试和故障排除技巧

#### 日志记录最佳实践

有效的日志记录对于排查集成问题至关重要：
1. **分级日志**：使用不同级别（INFO、WARNING、ERROR）记录信息
2. **结构化日志**：采用结构化格式，便于分析和查询
3. **上下文信息**：包含足够的上下文，如请求ID、操作类型、时间戳
4. **敏感信息处理**：避免记录密码、令牌等敏感信息
5. **性能考虑**：平衡日志详细程度和性能影响

#### 常见排错技巧
1. **二分法定位**：通过在流程中间点添加日志，快速缩小问题范围
2. **单节点测试**：隔离测试各个节点，确定具体故障点
3. **请求重放**：保存API请求，可以单独重放测试
4. **数据对比**：比对源系统和目标系统数据，找出差异
5. **监控指标**：关注性能指标和错误率变化，发现异常趋势
