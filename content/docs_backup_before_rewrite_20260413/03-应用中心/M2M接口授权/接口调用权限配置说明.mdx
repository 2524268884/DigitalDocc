## 概述

接口调用权限配置是专属集成平台的核心安全功能，支持对下游系统（M2M调用）进行精细化的字段级权限控制。管理员可以针对用户、部门、角色等主体，配置不同资源类型（如部门、角色）的可读/可写字段清单，确保接口调用时仅返回或允许操作被授权的字段信息，提升数据安全合规性。

该功能主要解决企业级应用中不同下游系统对数据访问权限的差异化需求，避免敏感信息泄露，同时支持灵活的权限策略配置和管理。

## 界面概览

接口调用权限配置界面位于应用配置模块中，通过新增的「M2M字段权限」标签页提供访问。界面采用分层设计：

![image.png](/images/接口调用权限配置说明/38df4802-793c-4516-8d68-7a9cc001bffc.png)

- **策略列表区域**：展示当前配置的所有权限策略
- **操作按钮区**：提供新增、编辑、删除策略的快捷入口
- **配置面板**：支持资源类型、操作类型、字段选择的组合配置

## 支持的操作类型

### 1. 查询操作

查询操作用于控制下游系统通过接口获取数据时的字段可见性。

**配置说明**：
- 选择「查询」操作后，系统将显示该资源类型在属性管理中的全部字段
- 配置为可查询的字段，接口返回值中会包含该字段
- 未配置的字段将在返回值中被过滤掉（返回 null 或直接省略）

**使用场景**：
- 保护敏感字段（如薪资、身份证号等）不被下游系统获取
- 根据下游系统需求，裁剪不必要的字段减少数据传输量

### 2. 新增操作

新增操作用于控制下游系统通过接口创建资源时可以写入的字段。

**配置说明**：
- 选择「新增」操作后，系统将显示该资源类型的全部字段
- 配置为可新增的字段，下游系统可以在请求中传入
- 未配置的字段将不被接受，系统会忽略或报错

**使用场景**：
- 限制下游系统只能写入部分字段，保证数据质量
- 防止下游系统设置敏感字段，保护数据安全

### 3. 修改操作

修改操作用于控制下游系统通过接口更新资源时可以修改的字段。

**配置说明**：
- 选择「修改」操作后，系统将显示该资源类型的非必填字段
- 必填字段默认选中且不可删除，确保数据完整性
- 配置为可修改的字段，下游系统可以在更新请求中传入
- 未配置的字段将不被接受修改

**注意事项**：
- 必填字段在修改操作中不可移除，如需调整请在属性管理中修改字段属性
- 字段选择支持搜索功能，可在下拉框顶部输入字段名称快速定位

### 4. 删除操作

删除操作用于控制下游系统通过接口删除资源的权限。

**配置说明**：
- 选择「删除」操作后，下游系统可以调用删除接口删除资源
- 可以按资源类型配置删除权限，实现精细化控制

**安全建议**：
- 删除操作权限应谨慎授予，仅给必要的下游系统
- 建议同时启用删除前校验和删除后审计

## 详细操作

### 新增M2M字段权限策略

#### 创建新的权限策略

用于为指定资源类型配置字段级访问权限，支持新增和修改操作的精细化控制。

![image.png](/images/接口调用权限配置说明/bf004c66-16d3-4865-ab06-8419abfa1c28.png)

1. 进入应用配置页面，点击「M2M字段权限」标签页
2. 点击「添加策略」按钮，弹出配置对话框
3. 在「选择资源」下拉列表中选择目标资源类型（用户、部门）
4. 在「选择操作类型」中选择操作类型：
   - **查询**：允许查询该资源类型的指定字段
   - **新增**：允许下游系统创建该资源类型的新记录
   - **修改**：允许下游系统修改该资源类型的现有记录
   - **删除**：允许删除该资源类型的记录
5. 在「选择允许的字段」多选框中勾选可操作的字段
6. 点击「保存」按钮完成策略创建

#### 编辑已有策略

1. 在策略列表中找到需要编辑的策略
2. 点击该策略右侧的「编辑」按钮
3. 修改配置信息
4. 点击「保存」按钮确认修改

#### 删除策略

1. 在策略列表中找到需要删除的策略
2. 点击该策略右侧的「删除」按钮
3. 在确认对话框中点击「确定」完成删除

**注意事项**：
- 所有策略配置操作将被审计记录，包括修改人和操作时间
- 删除策略后，下游系统的相关接口调用将不再受该策略约束

## API行为说明

### 查询接口的字段过滤

当配置了查询权限后，API的返回行为如下：

| 字段配置状态 | API 返回值 |
------------|-----------|
| 配置为可查询 | 返回该字段的实际值 |
| 未配置查询权限 | 返回 null 或省略该字段 |

**示例**：

假设用户资源有以下字段：`username`、`email`、`phone`、`salary`

配置查询权限为：`username`、`email`、`phone`（不包含 `salary`）

```json
// API 返回值
{
  "username": "zhangsan",
  "email": "zhangsan@example.com",
  "phone": "13800138000"
  // salary 字段被过滤掉
}
```

### 新增接口的字段校验

当配置了新增权限后，API的请求校验行为如下：

| 字段配置状态 | API 请求处理 |
------------|-----------|
| 配置为可新增 | 接受该字段的值 |
| 未配置新增权限 | 忽略该字段（不写入） |

**示例**：

配置新增权限为：`username`、`email`（不包含 `phone`）

```json
// 下游系统请求
{
  "username": "zhangsan",
  "email": "zhangsan@example.com",
  "phone": "13800138000"  // 该字段将被忽略
}

// 系统实际写入
{
  "username": "zhangsan",
  "email": "zhangsan@example.com"
  // phone 字段未被写入
}
```

### 修改接口的字段校验

当配置了修改权限后，API的请求校验行为如下：

| 字段配置状态 | API 请求处理 |
------------|-----------|
| 配置为可修改 | 接受该字段的值更新 |
| 未配置修改权限 | 忽略该字段（不更新） |

## 最佳实践

### 1. 最小权限原则

仅授予下游系统所需的最小字段权限，避免过度授权带来的安全风险。

### 2. 字段分组管理

根据业务需求，将常用字段组合创建为多个策略，便于管理和复用。

### 3. 定期审计

定期检查和清理不再使用的权限策略，确保权限的时效性。

### 4. 测试验证

修改权限策略后，使用测试账号验证接口行为是否符合预期。

## 常见问题

### Q1：修改权限后下游系统报错

**可能原因**：
- 下游系统请求了未授权修改的字段
- 权限修改前的缓存未更新

**解决方案**：
- 检查并更新下游系统的请求参数
- 确认权限配置已生效（可能需要等待缓存刷新）

### Q2：查询接口返回的字段与配置不符

**可能原因**：
- 配置了查询权限但接口未正确应用
- 存在多个策略冲突

**解决方案**：
- 检查是否存在其他策略覆盖了当前配置
- 确认资源类型和操作类型选择正确

### Q3：必填字段无法取消修改权限

**原因**：必填字段必须保持可修改状态，以保证数据的完整性和业务正常运行。

**解决方案**：
- 如需限制对必填字段的修改，先在属性管理中将该字段改为非必填
- 然后再配置修改权限策略
